## 1. Overview and Rationale

### Why Fine-Tune

Out-of-the-box VLMs achieve only ~11-15% accuracy on multi-class soccer action classification (SoccerNet benchmarks). Our current Qwen3-VL-32B pipeline confirms this: the VLM cannot reliably distinguish goals from saves at sideline camera distance (1-2 pixel difference in ball position relative to goal line).

Jiang et al. (CVPR 2025 Workshop, arxiv 2505.13860) demonstrated that curriculum fine-tuning of LLaVA-Next-Video on 20K soccer clips improved classification from 11.8% to 63.5% -- a 5.4x lift. Their approach:
- 3-stage curriculum: concept alignment, VQA instruction tuning, task fine-tuning
- Synthetic labels generated by Claude 3.5 Sonnet from ground-truth events + extracted frames
- 8 frames uniformly sampled from 2-second clips (0.5s before, 1.5s after event)
- LoRA applied to language model only, visual encoder frozen
- Hardware: 4x A10 GPUs (24GB each)

### Our Adaptation

We cannot replicate Jiang et al. exactly (they had 20K clips from SoccerNet broadcast footage; we have sideline camera). But we can apply the same curriculum learning pattern to our domain:

| Aspect | Jiang et al. | Our Pipeline |
|--------|-------------|--------------|
| Base model | LLaVA-Next-Video | Qwen3-VL-8B-Instruct |
| Camera | Broadcast (close-up) | Sideline (~50m, wide-angle) |
| Training clips | 20K from 550+ matches | ~4,500-5,000 from 19 matches |
| Label source | SoccerNet GT + Claude 3.5 Sonnet | Analytics platform GT (exact timestamps) |
| Frame sampling | 8 frames, 2s window | 8 frames, 4s window (wider for sideline) |
| Hardware | 4x A10 (96GB total) | 1x RTX 3090 (24GB) |
| Framework | Custom | ms-swift (official Qwen recommendation) |

### Target Task

The fine-tuned model replaces the general-purpose Qwen3-VL-32B in Phase 3 (VLM verification). It answers the same classification prompts but with domain-specific understanding of:
- What a goal vs save looks like from 50m away
- Post-event signals (celebrations, kickoff restarts, goal kicks)
- Sideline camera perspective (players appear small, ball often invisible)
- Set piece formations (corner, free kick, goal kick, throw-in)

### 2.2 Halftime Offset Problem

Ground truth timestamps (`event_time`) are in match clock (milliseconds from period start). Video timestamps are continuous with pre-game warmup and halftime break. We must compute offsets for each half independently.

**Offset calibration method** (proven in the Rush match calibration):

1. Parse `period_duration` from JSON (typically 2,700,000ms = 45 min per half)
2. Use the first event of each half as an anchor
3. For each video, detect the first whistle (audio) or first clearly visible event (throw-in, kickoff) and align to GT timestamp
4. Store offsets: `video_time = event_time + half_offset`

For the Rush match, calibrated offsets were:
- H1 offset: ~304s (5.1 min pre-game)
- H2 offset: ~1053s (includes halftime gap of ~749s / 12.5 min)

The extraction script computes these offsets semi-automatically (see Section 10).

### 2.3 Frame Sampling Strategy

**Decision: 8 frames from a 4-second window at 2 FPS.**

Rationale:
- Jiang et al. used 8 frames from 2s window (0.5s before, 1.5s after) on broadcast close-ups
- Our sideline footage needs wider temporal context because the action is harder to see: post-event signals (celebrations, restarts) are more diagnostic than the action itself
- 4-second window: 1.0s before event, 3.0s after event
- 2 FPS sampling yields 8 frames from 4 seconds
- This matches our production VLM verifier's `_CLIP_FPS = 2`

**Extended windows for specific event types:**

| Event Type | Window (before/after) | Frames | Why |
|------------|----------------------|--------|-----|
| Goal | 2s / 10s | 8 (0.67 FPS) | Need to capture celebration/kickoff restart |
| Save (catch) | 1s / 5s | 8 (1.3 FPS) | Need aftermath: GK holding ball, distribution |
| Save (parry) | 1s / 3s | 8 (2 FPS) | Ball deflection + corner/goal kick restart |
| Shot on/off target | 1s / 3s | 8 (2 FPS) | Standard window |
| Corner kick | 2s / 6s | 8 (1 FPS) | Full delivery cycle |
| Goal kick | 1s / 5s | 8 (1.3 FPS) | Setup through delivery |
| Throw-in | 1s / 3s | 8 (2 FPS) | Distinctive pose |
| Free kick | 2s / 4s | 8 (1.3 FPS) | Setup through delivery |
| Penalty | 3s / 8s | 8 (0.7 FPS) | Setup through result |

### 2.4 Frame Resolution

**768 pixels wide** (matching production `_CLIP_WIDTH = 768`).

At this resolution, each frame uses ~200 vision tokens in Qwen-VL. 8 frames = ~1,600 vision tokens per sample. With prompt text (~200 tokens) and response (~50 tokens), total per sample is ~1,850 tokens. This is well within the Qwen3-VL-8B context window and memory budget.

### 2.5 Extraction Script

The script `scripts/extract_training_data.py` (implementation below in Section 10):

1. Reads each game's JSON files (1st Half, 2nd Half)
2. Computes halftime offsets (manual anchors stored in a calibration file)
3. For each relevant event, computes the video timestamp
4. Extracts frames using FFmpeg at the configured window/FPS
5. Saves frames as JPEG files in a structured directory
6. Generates ms-swift JSONL training records

**Directory structure:**

```
training_data/

      goal_382440_f001.jpg

  train.jsonl

  calibration.json    # Per-game half offsets
  manifest.json       # Dataset statistics
```

## 3. Hard Negative Mining

### 3.1 Why Hard Negatives Matter

Without hard negatives, the model learns to always predict the most common class. With only positive examples, it never learns to say "this is NOT a goal" or "this is NOT a save."

Jiang et al. did not focus on hard negatives because they used the SoccerNet full taxonomy (17 classes, no "none" class). Our pipeline is different: the VLM must also handle "none" (normal play) since it verifies candidates from the motion detector.

### 3.2 Hard Negative Categories

**Category 1: Goal vs Save Confusion (highest priority)**

For every goal event, generate a paired hard negative:
- Extract frames from saves that occur within +/-5 minutes of a goal
- Label: "This is a SAVE, not a goal" (with reasoning about what distinguishes them)
- Key teaching signal: "no celebration, GK holds ball, no kickoff restart"

For every save event, generate a paired hard negative:
- Extract frames from goals that occur within +/-5 minutes of a save
- Label: "This is a GOAL, not a save" (with reasoning)
- Key teaching signal: "players celebrating, kickoff restart visible"

**Category 2: Shot Outcome Confusion**

Pair shots-on-target with shots-off-target:
- Shot on target: followed by save or goal
- Shot off target: followed by goal kick (ball went over/wide)
- Shot blocked: defender blocks, play may continue
- Teaching signal: the RESTART after the shot tells you the outcome

**Category 3: Normal Play (random negatives)**

Sample random 4-second clips from periods with no labeled events:
- Between any two consecutive events, if the gap is >30 seconds, sample the middle
- Label: "none" -- normal play, nothing significant
- Target: 1 "none" sample per 3 positive samples (25% of dataset)

**Category 4: Confusing Set Pieces**

- Corner vs free kick: both involve a stationary ball and a delivery into a crowded area
- Goal kick vs free kick: both involve a kick from near the goal, but goal kick is inside 6-yard box
- Corner vs goal kick: happen at opposite ends of the same event sequence (shot -> parry -> corner OR shot -> miss -> goal kick)

### 3.3 Balancing Strategy

Target distribution per training batch:

| Class | Positive Samples | Hard Negatives | None (random) | Total |
|-------|-----------------|----------------|---------------|-------|
| Goal | 42 | 42 (save clips labeled "not goal") | - | 84 |
| Save (catch) | 104 | 50 (goal clips labeled "not save") | - | 154 |
| Save (parry) | 39 | 20 (shot clips labeled "not save") | - | 59 |
| Shot on target | 141 | - | - | 141 |
| Shot off target | 128 | - | - | 128 |
| Shot blocked | 64 | - | - | 64 |
| Corner | 93 | 30 (free kick labeled "not corner") | - | 123 |
| Goal kick | 220 | - | - | 220 |
| Throw-in | 200 (capped) | - | - | 200 |
| Free kick | 150 | 30 (corner labeled "not free kick") | - | 180 |
| Penalty | 2 | - | - | 2 |
| None | - | - | 300 | 300 |
| **Total** | **~1,183** | **~172** | **~300** | **~1,655** |

**Upsampling for rare classes**: Goals (42 samples) will be augmented via temporal shifts (Section 6) to ~168 samples. Penalties (2) are too rare to train on -- keep as test-only.

**Downsampling for dominant classes**: Throw-ins (739) capped at 200. Goal kicks (220) kept as-is since they are important for the pipeline (goal kick after a shot = shot off target).

## 4. Training Data Format

### 4.1 Framework Choice: ms-swift

**ms-swift** (ModelScope SWIFT) is the officially recommended framework by Qwen for fine-tuning Qwen-VL models. It has:
- Native Qwen3-VL support with multi-image/video training
- Built-in LoRA, QLoRA, and full-parameter training
- Flash attention and padding-free optimizations
- A simple JSONL dataset format with `images`/`videos` fields
- Single-GPU training support (critical for our 1x RTX 3090 setup)

Alternatives considered:
- **LLaMA-Factory**: Supports Qwen3-VL but less integrated with Qwen's vision processing pipeline
- **Unsloth**: Good memory optimization but less battle-tested for multi-image VLM training
- **2U1/Qwen-VL-Series-Finetune**: Well-documented, good alternative if ms-swift has issues

### 4.2 ms-swift JSONL Format

Each training sample is one line of JSONL:

```jsonl
{"query": "<image><image><image><image><image><image><image><image>\nYou are analyzing 8 frames sampled at 2 FPS from a 4-second clip of a soccer match recorded by a SIDELINE CAMERA at ~50 metres from the field.\n\nClassify the MAIN event visible in these frames. Choose ONE:\n- \"goal\": A goal was scored — you see celebration (arms raised, group hugs) OR kickoff restart at center circle\n- \"save_catch\": Goalkeeper caught the ball — GK holding/cradling ball in hands after a shot\n- \"save_parry\": Goalkeeper parried/deflected the ball — followed by corner kick restart\n- \"shot_on_target\": Shot toward goal that was saved or scored\n- \"shot_off_target\": Shot that missed — followed by goal kick restart (ball went wide/over)\n- \"shot_blocked\": Shot blocked by a defender (not the GK)\n- \"corner_kick\": Ball placed at corner arc, kicked into the box\n- \"goal_kick\": Ball kicked from six-yard box after it went out over the goal line\n- \"throw_in\": Player holds ball overhead at sideline, throws it in\n- \"free_kick\": Ball placed on ground, kicked from a stoppage after a foul\n- \"penalty\": Penalty kick — one shooter vs goalkeeper from the penalty spot\n- \"kickoff\": Kick-off from center circle\n- \"none\": Normal play, nothing significant\n\nRespond with JSON: {\"event\": \"<type>\", \"confidence\": 0.0-1.0, \"reasoning\": \"brief explanation\"}", "response": "{\"event\": \"goal\", \"confidence\": 0.95, \"reasoning\": \"After the shot, players run toward each other with arms raised in celebration. The opposing team walks back toward the center circle for a kickoff restart.\"}", "images": ["training_data/frames/game_01/goal_382440_f001.jpg", "training_data/frames/game_01/goal_382440_f002.jpg", "training_data/frames/game_01/goal_382440_f003.jpg", "training_data/frames/game_01/goal_382440_f004.jpg", "training_data/frames/game_01/goal_382440_f005.jpg", "training_data/frames/game_01/goal_382440_f006.jpg", "training_data/frames/game_01/goal_382440_f007.jpg", "training_data/frames/game_01/goal_382440_f008.jpg"]}
```

**Key format details for ms-swift:**
- `images`: list of file paths, one per `<image>` token in the query
- `query`: the user prompt with `<image>` tokens matching the images list length
- `response`: the target output (what we want the model to learn to produce)
- No `system` field needed -- the classification instructions are in the query
- No `history` field needed -- single-turn classification

### 4.3 Video Frames as Image List

ms-swift supports passing video frames as a list of images in the `videos` field:

```jsonl
{"query": "<video>\nClassify the event...", "response": "...", "videos": [["frame1.jpg", "frame2.jpg", ..., "frame8.jpg"]]}
```

However, using `images` (one `<image>` token per frame) gives us more control over token allocation and matches how our production pipeline sends frames to vLLM. We use the multi-image format, not the video format.

### 4.4 Response Format

The response must match what we expect in production. Our current VLM verifier expects:

```json
{"event": "<type>", "confidence": 0.0-1.0, "reasoning": "brief explanation"}
```

The training response includes realistic reasoning that teaches the model what visual cues to look for:

| Event | Example Reasoning |
|-------|------------------|
| goal | "After the shot, players run toward each other with arms raised in celebration. The opposing team walks back toward the center circle for a kickoff restart." |
| save_catch | "The goalkeeper dives to their right, gathers the ball, and stands up holding it against their chest. Play pauses as the GK prepares to distribute." |
| save_parry | "The goalkeeper dives and deflects the ball wide. A corner kick is set up — a player walks toward the corner flag." |
| shot_on_target | "A player strikes the ball toward goal. The goalkeeper moves to make a save." |
| shot_off_target | "A player shoots toward goal but the ball goes over the crossbar. A goal kick is being set up." |
| corner_kick | "A player stands at the corner flag with the ball at their feet. Multiple players from both teams are gathered in and around the penalty area." |
| none | "Players are passing the ball in midfield. No shot, set piece, or significant event is occurring." |

### 4.5 Curriculum Learning (2 Stages)

Following Jiang et al., we use a simplified 2-stage curriculum:

**Stage 1: Concept Alignment (descriptive)**

The model learns to describe what it sees in soccer-specific terms:

```jsonl
{"query": "<image>...<image>\nDescribe what is happening in this soccer clip. Focus on: ball position, player formations, goalkeeper actions, celebrations, and restart patterns.", "response": "A player in white kicks the ball toward the goal from the left side of the penalty area. The goalkeeper in teal dives to their right and catches the ball, holding it against their chest. Other players stop running and begin walking back upfield. The goalkeeper stands up with the ball and looks to distribute it.", "images": [...]}
```

**Stage 2: Classification (task-specific)**

The model learns to classify using the exact production prompt format:

```jsonl
{"query": "<image>...<image>\nClassify the MAIN event...", "response": "{\"event\": \"save_catch\", \"confidence\": 0.95, \"reasoning\": \"...\"}", "images": [...]}
```

Stage 1 training data is generated by running the Qwen3-VL-32B base model on each clip and having it describe what it sees, then manually reviewing/correcting the descriptions. Alternatively (and faster), we write template descriptions per event type and vary them with synonyms.

## 6. Data Augmentation and Bootstrapping

### 6.1 The Goal Scarcity Problem

58 goals across 19 matches. This was the binding constraint but is now much healthier than the original 16-goal estimate. Saves (184), shots (459), and set pieces (1,720) have strong representation.

### 6.2 Temporal Shift Augmentation

For each goal event, create 3 additional training samples with shifted windows:

| Variant | Window | Teaching Signal |
|---------|--------|----------------|
| Original | -1s to +10s | Standard: shot + celebration |
| Early shift | -3s to +8s | More buildup, earlier in celebration |
| Late shift | +2s to +13s | Celebration + kickoff restart walk-back |
| Restart focus | +8s to +20s | Kickoff restart only (no shot visible) |

This yields 58 x 4 = **232 goal training samples** from 58 unique events.

Apply the same 4x augmentation to saves (catch only, since those are our hardest problem):
- 184 total saves x 4 = **~736 save training samples** (split between catch and parry)

For other classes, apply 2x augmentation (original + late shift):
- 39 parries x 2 = 78
- 141 shots on target x 2 = 282 (capped at 200)
- 93 corners x 2 = 186

### 6.3 Frame Jitter

For each augmented sample, apply random frame jitter: instead of extracting at exactly 2 FPS timestamps, offset each frame by +/-0.1s. This prevents the model from memorizing exact frame positions.

### 6.4 Brightness/Contrast Variation

Some matches are played in different lighting conditions (sunny, overcast, evening). Apply mild random brightness/contrast adjustments (+/-10%) to increase robustness.

### 6.5 Mirror (Horizontal Flip)

Soccer is symmetric. Flipping all frames horizontally doubles the effective dataset without changing the label. However, do NOT flip throw-ins (handedness matters for the pose) or corners (the corner flag position changes meaning).

### 6.6 Augmented Dataset Size

## 7. Evaluation Strategy

### 7.1 Holdout Set Design

**Hold out 2-3 complete matches** (game 11 = Rush vs Reign, game 13 = high-goal game, game 17 = 4 goals + penalty) as the evaluation set. These are chosen because:
- Game 11 has comprehensive ground truth already validated in our Rush pipeline runs
- Game 13 has the most goals (10) -- important for goal evaluation
- Game 17 adds diversity (penalty kick, different opponent)
- Together they cover 18 goals, 41 saves, 23 corners -- representative distribution

The remaining 16 matches are training data.

**Never augment the holdout set.** Evaluate on original clips only.

### 7.2 Metrics

**Primary metric**: Per-class accuracy (matching production use case)

| Metric | What It Measures |
|--------|-----------------|
| Overall accuracy | Correct predictions / total predictions |
| Per-class precision | TP / (TP + FP) for each event type |
| Per-class recall | TP / (TP + FN) for each event type |
| Goal precision | How many predicted goals are real (FP rate) |
| Goal recall | How many real goals are detected (FN rate) |
| Save precision/recall | Same for saves |
| Confusion matrix | Which classes are confused with which |

**Target metrics** (based on Jiang et al.'s results scaled for our smaller dataset and harder camera angle):

| Class | Baseline (current) | Target (fine-tuned) | Rationale |
|-------|-------------------|--------------------|-----------| 
| Goal | 25% (1/4) | 70-80% | Currently worst class, biggest improvement expected |
| Save | ~60% | 75-85% | Post-event signals learnable |
| Shot | ~80% | 85-90% | Already decent, incremental improvement |
| Corner | ~90% | 95%+ | Already strong |
| Goal kick | ~80% | 90%+ | Distinctive, learnable |
| Overall | ~50% | 70-80% | Jiang et al. achieved 63.5% on harder 13-class task |

### 7.3 A/B Testing in Production

After training, run both models on the same match and compare:

1. Run the production pipeline with the current Qwen3-VL-32B (baseline)
2. Run the same pipeline with the fine-tuned Qwen3-VL-8B (experiment)
3. Compare detected events against ground truth
4. Measure: precision, recall, F1 per event type

The fine-tuned 8B model should match or beat the 32B model on soccer-specific tasks despite being 4x smaller, because domain adaptation compensates for reduced general capability.

### 7.4 Regression Testing

Maintain a test suite of "known hard cases" from the Rush match:

| Case | Video Timestamp | Expected | Why It's Hard |
|------|----------------|----------|---------------|
| McCann goal | ~11:26 | goal | Close-range, minimal celebration |
| Rush goal from corner | ~68:10 | goal | From corner, classified as catch in v4 |
| Pruner penalty goal | ~71:56 | goal | Penalty kick format |
| Elena catch at 37:26 | ~37:26 | save_catch | Only catch VLM detected in v4 |
| Opening kickoff | ~5:00 | kickoff (not goal) | Pre-game kickoff confused with post-goal |
| Late match save | ~97:56 | save (not goal) | Post-game activity confused with goal |

## 8. Minimum Viable Dataset Analysis

### 8.1 Is 19 Matches Enough?

**Yes, more than enough for a strong first iteration.** Here is the evidence:

Jiang et al. tested with 3K, 10K, and 20K training samples:
- 3K samples: ~40% accuracy (vs 11.8% baseline)
- 10K samples: ~55% accuracy
- 20K samples: 63.5% accuracy

Our augmented dataset of ~4,500 samples from 19 matches maps between their 3K and 10K experiments. Expected accuracy: **50-60%** (our camera is harder, but our labels are higher quality than their synthetic Claude-generated labels, and we have more data than initially planned).

More importantly, for the specific goal-vs-save discrimination problem:
- 58 real goal events (augmented to ~232-464 with temporal shifts and mirrors) is a strong foundation
- 184 save events (augmented to ~500+) provides robust negative signal
- The model needs to learn ONE key pattern: "celebration/kickoff after shot = goal; GK holds ball / corner / goal kick after shot = save"

### 8.2 Expected Improvement by Dataset Size

| Matches | Goals | Augmented Samples | Expected Overall Accuracy | Expected Goal Accuracy |
|---------|-------|-------------------|--------------------------|----------------------|
| 6 (original plan) | 16 | ~1,200 | 35-45% | 50-60% |
| 13 | 42 | ~3,200 | 45-55% | 65-75% |
| **19 (current)** | **58** | **~4,500** | **50-60%** | **70-80%** |
| 30 (target) | ~100 | ~8,000 | 60-70% | 80-90% |

### 8.3 Recommendation

**Start training now with 19 matches.** We have significantly more data than the original 6-game / 16-goal estimate. The first iteration with 19 matches will:

1. Validate the entire pipeline (extraction, training, deployment)
2. Identify which classes need more data (may not be goals at all)
3. Likely beat the current 32B model on goal detection (from 25% to 70-80%)
4. Provide a baseline for measuring improvement as more matches are added

The fine-tuning is cheap (~3-4 hours on one RTX 3090) and the model can be retrained incrementally as more data arrives. With 58 goals, we are well past the minimum viable threshold.

## 9. Integration with Production Pipeline

### 9.1 Model Swap

The fine-tuned Qwen3-VL-8B replaces the current Qwen3-VL-32B in the VLM verifier:

**Before (current):**
- Model: `Qwen/Qwen3-VL-32B-Instruct-FP8` (both GPUs, tensor-parallel)
- Inference speed: ~8-12s per candidate
- VRAM: ~40GB (both RTX 3090s)
- YOLO runs on CPU during VLM inference

**After (fine-tuned):**
- Model: `Qwen/Qwen3-VL-8B-Instruct` + LoRA adapter (single GPU)
- Inference speed: ~3-5s per candidate (4x smaller model)
- VRAM: ~18GB (one RTX 3090)
- Second GPU available for YOLO or parallel VLM inference

### 9.2 Prompt Compatibility

The fine-tuned model is trained on the exact same classification prompt as `_CLASSIFY_SINGLE_PASS_PROMPT` in `vlm_verifier.py`. The response format (`{"event": "...", "confidence": ..., "reasoning": "..."}`) is identical. No code changes needed in the verifier beyond updating the model name.

One change: the classification categories in the prompt should match the training labels exactly. Add `save_catch` and `save_parry` as separate categories (currently both map to `shot_stop_diving`).

### 9.3 Fallback Strategy

Keep the Qwen3-VL-32B available as a fallback:
1. Primary: Fine-tuned 8B (fast, domain-specific)
2. Fallback: Base 32B (slower, general-purpose) -- for candidates where 8B confidence < 0.4
3. Claude API: Emergency fallback (costly, but available)

### 9.4 Continuous Improvement Loop

Every processed match generates more training data:
1. Pipeline detects events with the fine-tuned model
2. Human reviews events in the UI (already built at `/ui`)
3. Corrections are logged (accept/reject/reclassify)
4. Monthly: add reviewed matches to training set, retrain
5. Track accuracy trends: each retrain should improve
